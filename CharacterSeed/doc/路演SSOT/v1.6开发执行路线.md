## Step 1: 数据模型 — models.py 变更

**实施内容**：
- 新增 `World` 表：`id, name, core_worldview, created_at`
- 新增 `Scene` 表：`id, world_id(FK), name, scene_layer, scene_type, parent_scene_id(FK→self), description, initial_description, attributes_json, created_day, created_at`
- 新增 `SceneChange` 表：`id, scene_id(FK), growth_log_id(FK, nullable), change_type, description, change_details_json, day_number, created_at`
- `Character` 表新增两列：`world_id(FK→worlds), current_scene_id(FK→scenes)` + `short_term_goals(Text)`

**排序逻辑**：数据模型是所有后续步骤的前提。没有表，CRUD 无法编写，API 无法定义。放在第一步，且集中在一个 PR 中，因为三表共享外键依赖——分开提交会导致中间状态不可用。

**为什么 models.py 一步完成而非分步**：
- `Scene.parent_scene_id` 自引用 FK 与 `Character.current_scene_id` 依赖 `Scene.id`
- 分步会导致迁移脚本需要多次运行，增加运维复杂度
- SQLAlchemy 的 `Base.metadata.create_all()` 在单次调用中处理所有表更可靠

## Step 2: 迁移脚本 — db_migration.py v003

**实施内容**：
- 新增 `migration_v003_world_system()`：创建 worlds / scenes / scene_changes 三表 + Character 加列
- 幂等检查：`IF NOT EXISTS` 对每个表 / 每列
- 存量角色回填：为每个现有角色创建默认 World + 从 `current_state.location` 提取文本创建默认实际场景 + 设置 FK

**排序逻辑**：紧随 models.py。迁移脚本的单元测试可以验证 Step 1 的模型定义是否正确（表是否可创建、FK 是否生效）。

**存量回填策略**：
```
对每个无 world_id 的 character：
  1. INSERT INTO worlds (name) VALUES (角色名 + "的世界")
  2. INSERT INTO scenes (world_id, name, scene_layer='conceptual', scene_type='region', description) 
     VALUES (world_id, '起始之地', 'conceptual', 'region', world_setting 的前 200 字)
  3. INSERT INTO scenes (world_id, parent_scene_id, name, scene_layer='actual', scene_type='location', description)
     VALUES (world_id, 上面那条的 id, current_state.location 文本, 'actual', 'location', current_state.location)
  4. UPDATE characters SET world_id=?, current_scene_id=? WHERE id=?
```

## Step 3: 场景 CRUD — crud/world.py + crud/scene.py

**实施内容**：

`crud/world.py`（8 个函数）：
- `create_world(db, name, core_worldview)` → World
- `get_world(db, world_id)` → World | None
- `get_world_by_character(db, character_id)` → World | None

`crud/scene.py`（5 个函数）：
- `create_scene(db, world_id, name, scene_layer, scene_type, parent_scene_id, description)` → Scene
- `get_scene(db, scene_id)` → Scene | None
- `get_scene_path(db, scene_id)` → list[Scene]（从 current 向上遍历 parent 到根，返回有序路径）
- `get_adjacent_scenes(db, scene_id)` → list[Scene]（同一 parent 下的兄弟场景）
- `update_current_scene(db, character_id, new_scene_id)` → Character

`crud/scene_change.py`（2 个函数）：
- `create_scene_change(db, scene_id, growth_log_id, change_type, description, change_details_json, day_number)` → SceneChange
- `get_recent_changes(db, scene_id, limit=20)` → list[SceneChange]

**排序逻辑**：CRUD 层是 API 和管线调用的基础。先于 schema 定义——因为 CRUD 的返回值决定 schema 需要暴露哪些字段。

**为什么 `get_scene_path` 不用递归 SQL**：
- SQLite 不支持递归 CTE 的某些写法
- 应用层循环 `while scene.parent_scene_id: scene = get_scene(scene.parent_scene_id)` 最多 10 层，O(10) 次 DB 查询可接受
- 可在 `get_scene_path` 内部增加缓存：`dict[int, Scene]` 避免同一次请求中重复查询

## Step 5: Creation Prompt 扩展 — creation.txt

**实施内容**：
新增第 11-12 字段输出要求：

```
11. "core_worldview" - 核心世界观（200 字以内文本）
12. "initial_scenes" - 初始场景数组，每项含：
    - "name": 场景名称
    - "scene_layer": "conceptual" 或 "actual"
    - "scene_type": 场景类型（continent/kingdom/forest/town/tavern/cave/room...）
    - "description": 场景叙事描述
    - "parent_scene_ref": 同一响应中父场景的数组索引（根节点为 null）
    - 角色当前位置的实际场景默认作为 initial_scenes 的最后一条
13. "character_start_scene_ref": 角色初始所在场景在 initial_scenes 中的索引
```

**排序逻辑**：在 schema 校验和端点改造之前。prompt 决定了 LLM 产出什么格式的 JSON，schema 校验决定了如何验证它，端点决定了如何持久化——三者必须顺序对齐。

**为什么用 `parent_scene_ref` 索引而非名称匹配**：
- LLM 产出的场景名可能不一致（"暴风王国" vs "暴风城·王国"）
- 索引引用是确定性的，无歧义
- 校验层在解析 `parent_scene_ref` 时将其映射为实际的 `Scene.id`

## Step 6: Creation Schema 校验 — llm_service.py

**实施内容**：
`validate_creation_schema()` 新增 world 相关校验：

```
校验规则：
  ✓ core_worldview 为字符串，空则保底 "一个充满未知的世界"

  ✓ 每条 scene 必须有 name、layer（"conceptual"/"actual"）、type、description
  ✓ actual 场景的 parent_scene_ref 必须指向 conceptual 节点
  ✓ parent_scene_ref 必须引用数组内已出现的索引
  ✓ character_start_scene_ref 必须有效

**排序逻辑**：在 `create_character` 端点改造之前。端点依赖 `parsed_data` 中的 scene 数据，校验层保证数据质量。

## Step 7: create_character 端点改造 + World API — main.py

**实施内容**：

```

## Step 9: InteractionPipeline 事件入口 — interaction.py

**实施内容**：
- 现有 `run()` 方法（对话入口）保持不变
- 新增 `run_event(event, character, db)` 方法（事件入口）

核心流程：
```
run_event(event, character, db):
  1. 计算人格加权引导 → compute_personality_influence(personality, event.type)
  2. 组装 Director prompt（事件模式）→ 注入 scene_path + today_schedule + personality_influence
  3. Director.analyze() → emotion/goal/capabilities/event_attitude
  4. Actor.generate() → action/speech/expression/dialogue_pending
  5. 处理 capabilities：
     - 若 initiate_dialogue → 创建 dialogue_callback 事件
     - 若 modify_plan → 处理 plan_modifications（日程侧+目标侧）
     - 若 complete_event(X) → 将子类型写入 Event.result_json 上下文
  6. 返回 EventResponse（含 director_raw + actor_raw + capabilities_applied）
```

**排序逻辑**：在 advance_event 重写之前。`run_event()` 是纯管线逻辑，可以被单元测试独立验证（不依赖 HTTP）。advance_event 是它的 HTTP 包装层。

**为什么 Director/Actor 不新建文件**：
- 现有 `interaction.py` 已经有 Director + Actor 实现
- `run_event()` 复用相同的 Director/Actor 实例，仅 prompt 模板不同
- 分支判断放在 prompt 组装阶段（`if event_mode: template = event_template else: template = chat_template`）
- 这样改动最小，不引入新模块

## 关键技术决策速查

| # | 决策 | 考量 |
|---|------|------|
| 1 | Scene 两层 (conceptual/actual) | 按交互性质分野，不按行政层级 |
| 2 | World N:1 Character | 多角色共享世界，删除角色不销毁世界 |
| 3 | SceneChange 间接迭代 | 保留叙事因果链，description 为核心 |
| 4 | 废弃 EventResolver，复用 Director+Actor | 统一交互管线入口，议程事件=对话事件 |
| 5 | 能力集合白名单 | 每个能力对应确定实现路径，LLM 只做选择 |
| 6 | complete_event 5 子类型 | succeed/exceed/linger/fail/skip 覆盖行为谱系 |
| 7 | 人格加权单函数封装 | 纯计算、无副作用、注入 prompt 为引导非强制 |
| 8 | 短期目标为 long_term→schedule 桥梁 | 不独立，生命依附于日程 |
| 9 | Growth + 角色双迭代目标 | Growth 全量（每天结束），角色局部（每个事件） |
| 10 | modify_plan 扩展为日程+目标双侧面 | 同一能力操作两个关联数据结构 |
