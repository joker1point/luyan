## 四、Phase 1 — 世界系统具象化（1.5 天）

### 4.1 目标

将 `world_setting` 从一段静态文本升级为一个**可演化、可共享、可追溯**的结构化世界系统。核心交付：
- 独立持久化的 World 表（角色 N:1 归属，为多角色复用预留）
- 两层场景树——概念场景（世界观框架）与实际场景（角色可互动）
- `SceneChange` 迭代表——以叙事描述为核心，桥接硬编码与自由描述
- Creation 初始化世界 + Growth 通过 SceneChange 记录迭代世界

### 4.2 核心设计理念

#### 4.2.1 概念场景 vs 实际场景：交互性质的分野

场景树仅分两层，不是按行政层级划分，而是按**角色能否直接互动**区分：

| 维度 | 概念场景 (conceptual) | 实际场景 (actual) |
|------|----------------------|-------------------|
| 角色能否直接互动 | 否——"我在暴风王国"没有可操作性 | 是——"我在旅店大厅里和老板交谈" |
| 包含关系 | 概念套概念 + 概念套实际 | 实际场景**只能**归属于概念场景 |
| 迭代频率 | 低——仅在世界观扩展时（新大陆、王国覆灭） | 高——每次事件都可能改变 |
| 核心字段 | name + description + parent_scene_id（归属关系） | name + description + attributes_json（扩展属性） |
| 本质 | **世界观的拓展形态**——Creation `world_setting` 的结构化表达 | 角色的**可操作空间**——事件发生地、对话场景 |

**关键约束**：实际场景的 `parent_scene_id` 必须指向概念场景，不能指向另一个实际场景。概念场景可以嵌套概念场景来组织世界框架。

```
概念场景层级（世界观框架）：
  艾泽拉斯大陆 (conceptual)
    └─ 暴风王国 (conceptual)
         └─ 艾尔文森林 (conceptual)

实际场景（角色可互动，归属于概念场景）：
  狮王之傲旅店大厅  → parent: 闪金镇(conceptual)
  闪金镇广场        → parent: 闪金镇(conceptual)
  铁匠铺            → parent: 闪金镇(conceptual)
  水晶矿洞          → parent: 艾尔文森林(conceptual)
```

#### 4.2.2 SceneChange 迭代表：以叙事描述为核心

直接修改 `Scene.description` 会丢失叙事因果链——你无法知道"旅店为什么烧毁了"。`SceneChange` 表保留这个链：

```python
class SceneChange(Base):
    """场景迭代记录——以叙事描述为核心，桥接硬编码与自由描述"""
    __tablename__ = "scene_changes"
    id               = Column(Integer, primary_key=True)
    scene_id         = Column(Integer, FK("scenes.id"), nullable=False, index=True)
    growth_log_id    = Column(Integer, FK("growth_logs.id"), nullable=True)
    change_type      = Column(String(20), nullable=False)  # "character_driven" | "external"
    description      = Column(Text, nullable=False)        # 核心：叙事化的变化描述
    change_details_json = Column(Text)                     # 可选结构化详情
    day_number       = Column(Integer, nullable=False)
    created_at       = Column(DateTime, server_default=func.now())
```

**`change_type` 两分法**：

| 类型 | 触发者 | 示例 | 对后续行为的影响 |
|------|--------|------|----------------|
| `character_driven` | 角色的行动/选择 | "角色说服了旅店老板降价"、"角色打翻油灯引发火灾" | 直接改变角色的 reputation/mood/关系 |
| `external` | 世界/环境/他人 | "暴风雪封住了山路"、"一支商队抵达小镇" | 为角色创造新的选择空间或约束条件 |

这个区分不是给 LLM 加限制，而是给 Growth 下一次迭代提供**结构化上下文**——它知道"角色造成的后果"和"外界强加的变化"需要不同的应对策略。

**"硬编码与自由描述"之间的平衡**：

| 字段 | 性质 | 说明 |
|------|------|------|
| `scene_id` | 结构化（硬） | 精确指向受影响的场景 |
| `change_type` | 结构化（硬） | character_driven / external |
| `description` | **自由描述（软，核心）** | LLM 产出的叙事文本 |
| `change_details_json` | 软结构化 | 可选键值对，如 `{"new_npc": "流浪商人", "damage": "partial"}` |

**对 Scene 本体的更新策略**：SceneChange 创建时，钩子同步更新 `Scene.description` 为当前最新状态；`Scene.initial_description` 保留 Creation 时的原始描述作为不可变锚点。

### 4.3 数据模型总览

```python
class World(Base):
    """世界表 — 独立持久化，多角色可共享"""
    __tablename__ = "worlds"
    id = Column(Integer, primary_key=True)
    name = Column(String(100), nullable=False)        # 世界名称
    core_worldview = Column(Text, nullable=False)      # 核心世界观描述（Creation 产出）
    created_at = Column(DateTime, server_default=func.now())


class Scene(Base):
    """场景表 — 仅两层：概念场景(conceptual) + 实际场景(actual)"""
    __tablename__ = "scenes"
    id = Column(Integer, primary_key=True)
    world_id = Column(Integer, ForeignKey("worlds.id"), nullable=False, index=True)
    name = Column(String(100), nullable=False)
    scene_layer = Column(String(20), nullable=False)   # "conceptual" | "actual"
    scene_type = Column(String(30))                    # continent/kingdom/town/tavern/cave/...
    parent_scene_id = Column(
        Integer, ForeignKey("scenes.id"), nullable=True, index=True
    )
    description = Column(Text)                         # 当前场景描述（反映最新状态）
    initial_description = Column(Text)                 # Creation 时的原始描述（不可变锚点）
    attributes_json = Column(Text)                     # 扩展属性（heatmap_score/mood/danger_level...）
    created_day = Column(Integer, default=1)
    created_at = Column(DateTime, server_default=func.now())


class SceneChange(Base):
    """场景迭代记录表 — 以叙事描述为核心"""
    __tablename__ = "scene_changes"
    id = Column(Integer, primary_key=True)
    scene_id = Column(Integer, ForeignKey("scenes.id"), nullable=False, index=True)
    growth_log_id = Column(Integer, ForeignKey("growth_logs.id"), nullable=True)
    change_type = Column(String(20), nullable=False)   # "character_driven" | "external"
    description = Column(Text, nullable=False)          # 核心：叙事化的变化描述
    change_details_json = Column(Text)                  # 可选结构化详情
    day_number = Column(Integer, nullable=False)
    created_at = Column(DateTime, server_default=func.now())


# Character 表修改（相较于现有模型）
# world_id → worlds.id         ← 新增（N:1 共享世界）
# current_scene_id → scenes.id ← 新增（结构化位置，取代 current_state.location 纯文本）
```

### 4.4 三管线与世界系统的职责边界

```
Creation (初始化世界)
  → 创建 World 行（core_worldview）
  → 创建初始概念场景树（至少覆盖角色的完整上下层级链）
  → 创建角色起始实际场景
  → 设置 character.world_id + character.current_scene_id

Growth (迭代世界 — 唯一写入者)
  → 新增实际场景（角色发现新地点） → 同时创建 Scene + SceneChange 记录
  → 极少数新增概念场景（世界观扩展）→ 同时创建 Scene + SceneChange 记录
  → 创建 SceneChange 记录（场景变化叙事）→ 钩子同步更新 Scene.description
  → 极少：更新概念场景的归属关系（王国覆灭等）
  → 更新 character.current_scene_id（角色移动）

Interaction (只读)
  → 读取 current_scene 及其 parent 概念场景链
  → 读取最近 N 条 SceneChange 记录（叙事上下文）
  → 读取相邻实际场景列表（角色可选行动空间）
  → 不修改任何世界数据
```

**为什么 Growth 通过 SceneChange 间接更新而不是直接修改 Scene 行？**
- 保留完整叙事因果链——"旅店为什么烧毁"是可追溯的
- `description` 承载叙事（自由），`change_details_json` 承载结构化元数据（硬）——两者分离但共存
- 后续可实现"回滚"——撤销某个 SceneChange 即可恢复到之前的状态

### 4.5 Creation 初始化世界的策略——全量骨架 + 局部细节

Creation LLM 需要产出：
1. `core_worldview`（核心世界观文本，200 字内）
2. 初始概念场景树——至少包含角色当前位置的**完整上下层级链**
3. 角色初始实际场景

**原则：不需要产出整个世界的所有场景**——那是 Growth 的职责。Creation 只负责角色初始可见范围内的世界骨架。

```
例如 role 初始位置 = "狮王之傲旅店大厅"
Creation 产出：
  core_worldview: "这个世界由人类王国与精灵森林构成..."
  initial_conceptual_scenes: [
    {name: "艾泽拉斯大陆", layer: "conceptual", type: "continent", desc: "..."},
    {name: "暴风王国", layer: "conceptual", type: "kingdom", desc: "...", parent_ref: 0},
    {name: "艾尔文森林", layer: "conceptual", type: "forest", desc: "...", parent_ref: 1},
    {name: "闪金镇", layer: "conceptual", type: "town", desc: "...", parent_ref: 2},
  ]
  initial_actual_scenes: [
    {name: "狮王之傲旅店大厅", layer: "actual", type: "tavern_hall", desc: "...", parent_ref: 3},
  ]
  character_start_scene_ref: 0  (指向 actual_scenes[0])
```

`parent_ref` 用数组索引引用同一响应中之前的场景，避免 LLM 需要生成 UUID。

### 4.6 Growth 迭代世界的策略——SceneChange 驱动

Growth prompt 新增输入和输出：

**新增输入**（注入到 growth.txt prompt）：

**概念场景的迭代频率控制**：在 prompt 中明确指令——

```
概念场景（如大陆、王国、森林）的迭代频率极低。仅当故事中出现真正的
"世界观扩展"时才修改概念场景——如发现新大陆、王国覆灭、区域归属改变。
日常迭代只影响实际场景（tavern/room/cave等角色可直接互动的地点）。
```

### 4.8 关键校验规则

| 规则 | 校验位置 | 动机 |

## 五、Phase 2 — 交互管线统一事件入口（1.5 天）

### 5.1 核心洞察

**日程事件和玩家对话本质上是同一种东西——对角色的"输入刺激"**。"玩家说你好"是一个刺激，"清晨练剑的时间到了"也是一个刺激。角色对两者的处理流程应该是统一的：Director 理解上下文并做出决策 → Actor 生成具体的响应/行为叙事。

当前设计有两个架构错误：
1. 为日程事件创建了独立的轻量 `EventResolver` LLM——这割裂了交互管线，导致日程事件的叙事质量远低于对话
2. Director 只用于对话场景，而日程事件的"角色决策"（接受/拒绝/拖延/搞砸）实际上需要同一个 Director 来做——只有他了解角色的全部上下文

**修正**：删除 `EventResolver`，将日程事件作为 `InteractionPipeline` 的第二个入口。Director + Actor 双层级联统一处理所有类型的事件。

### 5.2 统一入口设计

```
InteractionPipeline
  ├─ 对话入口（已有）
  │   输入: user_message → Director.analyze() → Actor.generate() → ChatResponse
  │
  └─ 事件入口（新增）
      输入: pending Event.content → Director.analyze_event() → Actor.generate_event() → EventResponse
```

两个入口共享同一套 Director 和 Actor 实例，区别仅在输入格式和 Director 的 prompt 侧重：

| 维度 | 对话入口 | 事件入口 |
|------|---------|---------|
| 输入刺激 | 玩家消息文本 | 事件描述文本 + 角色当天的完整日程表 |
| Director 核心决策 | 情绪 + 对话目标 + 记忆聚焦 | **能力选择**（respond/initiate/modify/complete）+ 情绪 + 事件态度 |
| Actor 输出 | speech + action + expression | action（叙事化行为描述）+ speech（如有对话对象）+ expression |
| 持久化 | Conversation 表 | Event.result_json + Event.status 更新 + 可能的同时创建对话回调 |

### 5.3 角色能力边界定义

角色面对事件时的决策能力是一个**有界能力集合**（Capability Set）。Director 必须知晓这些能力的边界并在其子集中做出选择：

```
┌─────────────────────────────────────────────────────────────┐
│                    角色事件能力集合 (Capability Set)           │
│                                                               │
│  respond_normally   正常回应事件（完成事件并给出叙事化描述）     │
│    ├─ 默认能力，所有角色在任何事件下都可选择                     │
│    └─ Actor 输出：事件执行的行为描述 + 情绪/表情                │
│                                                               │
│  initiate_dialogue  主动发起对话                               │
│    ├─ 事件内容触发角色想对玩家说话（"看到夕阳想起了往事想分享"）  │
│    ├─ 可附带 timed_dialogue_callback 创建定时回调               │
│    │   → 在当前事件完成后，自动将"角色想说的内容"包装为           │
│    │      一次反向对话事件，到期时触发 Director+Actor 生成对话    │
│    └─ 模拟角色主动找玩家说话的效果                               │
│                                                               │
│  modify_plan        基于短期目标调整日程                         │
│    ├─ 角色认为当前事件与其短期目标不符或优先级更低                │
│    ├─ 修改当前事件 content + 重新排队（或降优先）                │
│    ├─ 可同时为今天新建一条更相关的事件                           │
│    └─ 需要 Director prompt 中注入 short_term_goals 供其评估      │
│                                                               │
│  complete_event     结束事件（5 种子类型）                       │
│    ├─ succeed  正常完成 — 角色按预期完成事件                     │
│    ├─ exceed   超出预期 — 做得极其出色，结果超原始描述            │
│    ├─ linger   拖延敷衍 — 拖了很久或做得不彻底                   │
│    ├─ fail     尝试但失败 — 尝试了但搞砸了                       │
│    └─ skip     跳过不做 — 角色认为不重要，根本不去做              │
│         └─ 子类型选择：由 `compute_personality_influence()` 函数   │
│              计算人格加权倾向 → 注入 Director prompt 作为引导      │
│              → Director 做最终选择（能力集合的子集）               │
│              函数为纯数学计算，无 LLM 调用，无外部依赖             │
└─────────────────────────────────────────────────────────────┘
```

**为什么能力集合采用"子集选择"而非"开放决策"？**
- 防止 LLM 产出系统无法执行的动作（如"角色决定离开这个世界"）
- 每个能力对应后端具体的实现路径（创建对话回调、修改事件内容等）
- Director 只需从预定义集合中选出最合适的一项或多项组合（如 `initiate_dialogue + complete_event(succeed)`）

### 5.3.1 人格加权函数 — `compute_personality_influence()`

整套人格加权逻辑封装在 `backend/modules/interaction.py` 中的一个独立函数内，无外部依赖，在调用 Director LLM 之前运行。

**函数签名**：
```python
def compute_personality_influence(
    personality: dict[str, int],      # {"courage": 75, "intelligence": 68, ...}
    event_type: str,                   # "schedule_action" | "scene_event" | ...
) -> str:                              # 返回自然语言引导文本，直接注入 Director prompt
```

**加权计算逻辑**（纯数学，不调用 LLM）：

```python
def compute_personality_influence(personality, event_type):
    # 提取人格值（默认 50）
    c = personality.get("courage", 50)       / 100.0
    i = personality.get("intelligence", 50)  / 100.0
    s = personality.get("sociability", 50)   / 100.0
    e = personality.get("empathy", 50)       / 100.0
    l = personality.get("loyalty", 50)       / 100.0
    o = personality.get("optimism", 50)      / 100.0

    # ── complete_event 子类型权重 ──
    succeed_w = 0.30 + c*0.25 + i*0.20
    exceed_w  = 0.08 + i*0.28 + o*0.15       # 智力+乐观 → 超出预期
    linger_w  = 0.18 - c*0.18 + (1.0-s)*0.10 # 低勇气+低社交 → 拖延
    fail_w    = 0.12 - i*0.18 - c*0.10       # 低智力+低勇气 → 搞砸
    skip_w    = 0.12 - l*0.22 - e*0.12       # 低忠诚+低同理心 → 跳过

    # 归一化
    total = succeed_w + exceed_w + linger_w + fail_w + skip_w
    succeed_w, exceed_w, linger_w, fail_w, skip_w = [
        w/total for w in [succeed_w, exceed_w, linger_w, fail_w, skip_w]
    ]

    # ── 其他能力偏置 ──
    dialogue_bias = 0.25 + s*0.06 + o*0.04   # 社交+乐观 → 主动说话
    modify_bias   = 0.15 + i*0.04 - l*0.03   # 智力-忠诚 → 修改计划

    # ── 生成自然语言引导 ──
    dominant = max(
        [("succeed", succeed_w), ("exceed", exceed_w),
         ("linger", linger_w), ("fail", fail_w), ("skip", skip_w)],
        key=lambda x: x[1]
    )

    return (
        f"人格倾向分析（由系统计算，仅作引导参考）：\n"
        f"  你的性格特征：勇气{c*100:.0f} 智力{i*100:.0f} "
        f"社交{s*100:.0f} 同理心{e*100:.0f} 忠诚{l*100:.0f}\n"
        f"  事件完成倾向：{dominant[0]} 是最可能的结果（"
        f"succeed={succeed_w:.0%} exceed={exceed_w:.0%} "
        f"linger={linger_w:.0%} fail={fail_w:.0%} skip={skip_w:.0%}）\n"
        f"  主动对话倾向：{'高' if dialogue_bias > 0.45 else '中' if dialogue_bias > 0.3 else '低'}\n"
        f"  修改计划倾向：{'高' if modify_bias > 0.3 else '低'}\n"
        f"\n注意：以上仅为统计分析，你仍需根据当前事件的具体内容做出最终判断。"
        f"你有权选择与倾向不符的能力——只要你的决策有合理解释。"
    )
```

**调用位置**：`InteractionPipeline.run_event()` 中，Director prompt 组装前：

```python
influence_text = compute_personality_influence(personality, event.event_type)
prompt = director_event_template.format(
    ...,
    personality_influence=influence_text,
    ...
)
```

**设计约束**：
- 函数为纯计算——不接受 DB Session、不调用 API、不读取文件
- 全部逻辑在一个函数内，不超过 50 行
- 返回值为可直接注入 prompt 的字符串（非结构化 dict）
- 函数不替代 Director——它只提供倾向引导，Director 保有最终决策权
- 若 personality dict 缺少某维度，用默认值 50（中性）填充

### 5.4 Director 的事件入口 Prompt 结构

```
[事件模式] 角色正在处理一个日程事件

角色当前状态：
  姓名：{name}，当前心情：{mood}，当前位置：{scene_full_path}
  今日全部日程：{today_full_schedule}          ← 角色可见全部日程
  当前事件：[{event_type}] {event_content}    ← 正在处理的事件
  短期目标：{short_term_goals}                ← 供 modify_plan 参考

{personality_influence}                       ← 人格加权函数输出（计算所得引导文本）

可用能力集合（Director 必须在此集合内选择，可组合多个）：
  - respond_normally: 正常完成事件，生成叙事化行为描述
  - initiate_dialogue: 主动向玩家发起对话（可选 timed_callback 在完成后触发）
  - modify_plan: 修改当前或今日后续事件（需给出修改后的 event_content）
  - complete_event: 事件结束方式，必须选择以下之一：
      succeed / exceed / linger / fail / skip

人格加权引导仅作参考——你仍需根据当前事件的具体性质做出最终判断。例：
  低智力角色面对"解谜"类事件，即使 succeed 权重高，你也应考虑 fail 更合理。

Director 输出 JSON：
{
  "emotion": "专注",
  "goal": "完成日常训练保持剑术水平",
  "focus_memories": [{记忆...}],
  "style": "严肃认真",
  "capabilities": ["respond_normally", "initiate_dialogue", {"complete_event": "succeed"}],
  "dialogue_intent": "练剑结束后想和玩家聊聊今天的训练心得",
  "dialogue_callback_when": "after_event", 
  "plan_modifications": [],
  "event_attitude": "这个日常训练很重要，我会认真完成"
}


Actor 输出 JSON：
{
  "action": "拔出长剑，在晨曦中开始每日的基础剑术练习。剑光划破薄雾，每一次挥击都精准有力。一个时辰后，她收剑入鞘，额头渗出细密的汗珠。",
  "speech": null,                    // 无对话对象时可为 null
  "expression": "专注且满足",
  "dialogue_pending": {              // 仅当 Director 选了 initiate_dialogue
    "content": "今天的训练很顺利，感觉比昨天更进一步了。冒险者，你想试试和我对练吗？",
    "trigger_after_event": true,
    "trigger_time_period": "afternoon"
  }
}
```

### 5.5 日程可见性与主动对话的定时回调

**角色对当日日程的可见性**是 make-or-break 设计。如果角色不知道今天还有什么安排，`modify_plan` 就毫无意义。

```
Director 输入中包含的 today_full_schedule：
  [pending  #1 上午] 清晨练剑
  [pending  #2 下午] 去铁匠铺取回修理的剑  ← 当前正在处理
  [pending  #3 傍晚] 参加镇上晚宴
  [completed #0] 与冒险者进行了愉快的对话
```

角色看到了全天安排，Director 可以做出如下决策：
- "下午取剑很无聊，我改到傍晚吧，反正铁匠铺离晚宴场地很近" → `modify_plan`
- "傍晚晚宴是个社交好机会，我现在应该提前准备" → `complete_event(succeed)` + 修改后续事件

**定时对话回调** (`timed_dialogue_callback`)：

```
Director 输出: initiate_dialogue + dialogue_callback_when: "after_event"
  → 当前事件完成后，自动创建一条 character_initiative Event
  → 该事件的 content = "角色想和你说话：今天的训练很顺利..."
  → 当用户推进到该事件时，InteractionPipeline 的对话入口自动触发
  → 用户看到的是"角色主动走向你，说：冒险者，想试试对练吗？"

备选：dialogue_callback_when: "today_evening"
  → 在当天傍晚时段的事件推进前，插入一条对话回调事件
```

### 5.7 关键决策

#### 决策 1：废弃 EventResolver，复用 InteractionPipeline

**理由**：
- 日程事件和玩家对话是同一种"角色刺激"——Director 负责理解上下文并决策，Actor 负责生成叙事
- 割裂为两个管线导致日程事件的叙事质量远低于对话（小 LLM 无法生成丰富描述）
- Director 是唯一了解角色完整上下文（人格、记忆、当前场景、短期目标）的组件——只有他能做出有意义的"角色决策"
- 复用现有 Director+Actor 减少新增代码量，降低维护成本

#### 决策 2：能力集合作为有界白名单而非开放式决策

**理由**：
- 每个能力对应一个具体的后端实现路径——`initiate_dialogue` 创建 callback 事件，`modify_plan` 调用 `update_event_content()`
- 白名单确保 LLM 不会产出系统无法执行的动作（"角色离开了这个世界"→ 不可执行）
- Director 可在一次决策中组合多个能力（如 `initiate_dialogue + complete_event(succeed)` = "顺利完成任务并想和玩家分享"）
- 人格加权决策而非人格决定决策：高勇气只是提高 succeed 概率，不应绝对禁止 linger

#### 决策 3：`complete_event` 的子类型选择由人格加权而非 LLM 自由决策

**理由**：
- `succeed/linger/fail` 三种完成方式影响 `result_json` 的叙事质量，但不影响事件是否被标记为 completed
- 失败≠删除事件——失败的事件仍然是 completed，但其 result 反映了失败，供 Growth 迭代观察
- 人格加权算法：`P(succeed) = base + courage_weight + intelligence_weight`；最终选择是加权随机而非 LLM 直接输出——降低 LLM 滥用 fail 的风险

#### 决策 4：定时对话回调作为模拟角色主动性的轻量方案

**理由**：
- 真正的事件驱动主动对话需要服务端定时轮询——这在 MVP 阶段过度设计
- `timed_dialogue_callback` 创建一个 `character_initiative` 事件，用户推进到它时才触发对话
- 效果等价——用户看到的都是"角色主动找你说：..."
- 为后期升级到真实定时回调预留接口（Events 表已有 `time_period` 字段）

#### 决策 5：前端同时展示 Director 决策和 Actor 叙事

**理由**：
- 用户需要看到角色**为什么**做出某种反应（Director 决策卡片：情绪、态度、能力选择）
- 用户需要看到角色**具体做了什么**（Actor 叙事卡片：行为描述、表情、可能的对话）
- 这与当前对话气泡中已有的 Director 摘要卡片 + Actor 标签的设计完全一致
- 日程事件的 Actor 输出可能没有 speech（无对话对象），此时卡片只展示 action + expression

## 六、Phase 3 — 短期目标系统（0.5 天）

### 6.1 定位：长期目标与日程之间的桥梁

短期目标不是一个独立系统——它的存在意义是将抽象的长远追求分解为日程可执行的具体步骤。

```
长期目标（抽象，可能持续 30+ 天）       "成为天下第一剑客"
       ↓ 分解
短期目标（桥接，每条约 3-10 天）        "找到一位剑术大师指点"
                                        "挑战本地剑术冠军证明自己"
                                        "获得一把传说中的名剑"
       ↓ 驱动
日程事件（具体，每天 3-5 条）           Day3上午: 去酒馆打听剑术大师的消息
                                        Day3下午: 前往剑术道场拜访
                                        Day4上午: 挑战本地冠军 · 第一回合
```

短期目标的生命依附于日程——一条目标在多天产生一系列关联事件，全部事件完成（或角色主动放弃）时，这条目标才算完結。它不是独立运转的，而是通过日程事件来"活"的。

### 6.2 数据模型——极简 JSON 数组

```python
# Character 表新增
short_term_goals = Column(Text, nullable=True)
# JSON 数组，每条：
# {
#   "goal": "找到一位剑术大师指点",   # 目标描述
#   "progress": 0.30,                 # 0.0-1.0，基于关联事件完成比例
#   "created_day": 2,                 # 目标创建于第几天
#   "source": "growth"                # "creation" | "growth" | "character"
# }
```

**为什么不做独立表**：
- MVP 阶段不超过 5 条目标，JSON 列无性能问题
- 目标在任何管线中都需要与角色一起加载，独立表增加 JOIN
- `source` 字段标记创建者，区分 Creation 初始 / Growth 生成 / 角色自创的目标
- `created_day` 使 Growth 能计算一条目标已经持续了多少天（超过 N 天未进展 → 提示是否放弃）

### 6.3 双迭代者：Growth 和角色各司其职

短期目标有两个迭代者，但方式完全不同：

| 迭代者 | 时机 | 粒度 | 操作 |
|--------|------|------|------|
| **Growth** | 每天结束时（`/api/time/iterate`） | 大步：全量分析 | 基于当天所有完成事件，更新全部目标的 progress、标记完成、根据 long_term_goal 生成新目标 |
| **角色** | 每个事件执行时（`/api/event/advance`） | 小步：局部调整 | 基于当前事件的结果，微调一条目标的 progress；偶尔新增临时目标；偶尔放弃一条目标 |

**Growth 迭代（上帝视角）**：
```
输入：long_term_goal + 当前 short_term_goals + 当天所有 completed 事件
输出：{
  goal_updates: [
    {index: 0, new_progress: 0.65, reason: "今天拜访了道场，得到了大师的线索"},
    {index: 0, completed: true, reason: "大师正式收她为徒"},
  ],
  new_goals: [
    {goal: "修习大师传授的'流云剑法'第一式", source: "growth"}
  ],
  abandon_goals: []   // Growth 不主动放弃目标
}
次日 schedule 必须至少 1 条事件与 active_goals 中的某一条对齐。
```

**角色迭代（第一人称，属于 `modify_plan` 能力）**：
```
角色在事件执行时，通过 modify_plan 能力调整目标：
  → 完成了成功的事件 → progress 微增（+0.1 ~ +0.2）
  → 事件失败了 → progress 不变，但可能新增"弥补"类临时目标
  → 觉得目标太难 → 标记 abandon（格位保留，Growth 稍后可重新激活或替换）
  → 从 NPC 处获得线索 → 新增一条派生目标
```

### 6.4 `modify_plan` 能力范围扩展

`modify_plan` 现在包含对日程的修改和对目标的微调——两者是同一个能力的两个侧面：

```
modify_plan 完整能力范围：
  日程侧：
    1. 修改当前事件 content（原有）
    2. 调整事件 order_index / time_period（原有）
    3. 新增一条 pending 事件（原有）
  目标侧（新增）：
    4. 更新一条目标的 progress（±0.1 ~ ±0.3，Director 判断事件结果对目标的影响）
    5. 放弃一条短期目标（标记 abandoned，Director 给出原因）
    6. 新增一条临时短期目标（character-sourced，Director 给出原因）

Director 在事件模式下输出 plan_modifications 的结构扩展：
  "plan_modifications": [
    {"type": "event_reorder", "event_id": 12, "new_order_index": 3},
    {"type": "goal_progress", "goal_index": 0, "delta": 0.15, "reason": "..."},
    {"type": "goal_abandon", "goal_index": 1, "reason": "找到大师太困难了..."},
    {"type": "new_goal", "goal": "先从基本功练起", "reason": "..."}
  ]
```

## 八、关键决策与优先级考量

### 决策 1：Phase 1 先于 Phase 2

**理由**：World 模型是后续所有 LLM 行为的基础输入。如果在没有结构化 World 的情况下做 EventResolver，LLM 仍然不知道"角色在哪里、天气如何、有什么可用资源"，叙事质量不会提升。先建立世界坐标系，再做事件解析。

### 决策 2：废弃独立 EventResolver，日程事件复用 InteractionPipeline

**理由**：
- 日程事件和玩家对话是同一类"角色刺激"——Director 理解上下文并决策，Actor 生成叙事
- 割裂为两个管线导致日程事件叙事质量远低于对话，且角色决策缺少完整上下文
- Director 是唯一能访问角色完整上下文（人格、记忆、当前场景、短期目标）的组件
- 复用现有管线减少新增代码量，增加日程事件叙事深度

### 决策 3：角色能力集合为有界白名单 + 人格加权函数单一封装

**理由**：
- Director 的能力选择空间是预定义的——`respond_normally / initiate_dialogue / modify_plan / complete_event`
- `complete_event` 扩展为 5 种子类型：`succeed / exceed / linger / fail / skip`——覆盖人类从出色到放弃的完整行为谱系
- 每种能力对应一个确定的后端实现路径，LLM 不能产出系统无法执行的动作
- 人格加权逻辑封装在单一函数 `compute_personality_influence()` 内：纯数学计算（无 LLM 调用、无 DB 依赖），在调用 Director 前运行，返回自然语言引导注入 prompt
- **函数不替代 Director**——它只提供"人格倾向建议"；Director 有权选择与倾向不符的能力（只要决策有合理解释）
- `exceed`（超出预期）让高智力角色偶尔创造惊喜，`skip`（跳过不做）是低忠诚角色的标志性行为——两者补充了原 3 分类缺失的重要行为模式

### 决策 4：短期目标是长期目标→日程的桥梁，非独立系统

**理由**：
- 短期目标的存在意义是将 `long_term_goal`（抽象，"成为天下第一剑客"）分解为日程可执行的具体步骤（"找到大师→挑战冠军→获得名剑"）
- 目标的生命依附于日程——一条目标在多天产生关联事件，事件全部完成时目标自然完成
- 两个迭代者各司其职：Growth 每天结束做"上帝视角"全量分析（更新全部目标、生成新目标），角色每个事件做"第一人称"微调整（改 progress、放弃或新增目标）
- 角色的 `modify_plan` 能力扩展为同时操作日程（侧 1-3）和目标（侧 4-6），两者是同一能力的两个侧面
- 数据极简：JSON 数组，每项 `{goal, progress, created_day, source}`，无独立表

### 决策 5：World 独立持久化，Character N:1 归属

**理由**：
- `worlds` 表不含 `character_id`，由 `characters.world_id` FK 指向 World
- 独立持久化使得多个角色可以共享同一个世界——每个角色有自己的 `current_scene_id`，但共享同一棵场景树
- 删除角色不影响世界数据，除非是该世界中最后一个角色
- 这为 v1.2 多角色世界提供了自然的扩展路径：新角色加入现有世界时，只需 `character.world_id = existing_world.id`

### 决策 6：场景树严格两层——conceptual / actual

**理由**：
- 概念场景的颗粒度不足以让角色直接互动（"我在暴风王国"没有可操作性）
- 实际场景是角色行为的空间锚点（"我在旅店大厅和老板交谈"）
- 两层而非多层避免了"旅店大厅 → 旅店 → 闪金镇"这种无意义的实际场景嵌套
- 概念场景嵌套概念场景以组织世界观（大陆→王国→区域），但深度由 LLM 自然产出而非预设固定层数
- `scene_type` 是开放白名单（continent/kingdom/town/tavern/cave...），不是行政层级

### 决策 7：Growth 通过 SceneChange 间接迭代世界

**理由**：
- 直接修改 Scene 行会丢失叙事因果链——你无法追溯"旅店为什么烧毁了"
- SceneChange 以 `description` 为核心（自由叙事），`change_details_json` 为辅助（结构化详情），桥接了硬编码与自由描述之间
- `change_type` (character_driven / external) 的两分法让 Growth 的下一次迭代能区分"角色造成的后果"和"外界强加的变化"
- SceneChange 创建钩子同步更新 Scene.description（保持当前状态可读），而 Scene.initial_description 保留创建时的原始描述作为不可变锚点

### 决策 8：Phase 3 与 Phase 2 可并行

**理由**：
- Phase 2 改动集中在 `interaction.py`（事件入口 + Director 双模式）+ `main.py`（advance_event 重写）
- Phase 3 改动集中在 `prompts/growth.txt` + `growth.py` + `creation.txt`
- 两者无文件冲突，但 Phase 3 的 Growth prompt 需要引用 World 状态（Phase 1 产出）

## 十一、风险与预案

| 风险 | 影响 | 概率 | 预案 |
|------|------|------|------|
| LLM 解析事件时超时（>3s） | 用户体验差 | 中 | 设置 2s timeout + 回退到硬编码保底文案；自动模式可跳过 LLM 用快速路径 |
| Scene 树数据过长撑爆 prompt token | Growth 推理失败 | 低 | 只注入当前场景 + parent 链（最多 6 个节点）+ 相邻场景限 5 条 + 最近 SceneChange 限 3 条 |
| Creation 产出的 parent_ref 索引越界 | 角色创建崩溃 | 低 | schema 校验层验证所有 parent_ref < 数组长度，超界丢弃引用（节点变为根节点）+ warning 日志 |
| short_term_goals 保底为空 | 退化为当前行为 | 低 | Creation 保底 1 条通用目标（如"探索周围环境"） |
| 前端卡片过多导致页面冗长 | 信息过载 | 中 | 折叠面板（expander）+ Tab 切换；世界卡片默认折叠，点击展开 |
| 存量角色无 World/Scene 数据 | 旧角色无法使用新功能 | 中 | 迁移脚本 v003 自动为所有存量角色创建默认 World + 从 current_state.location 回填 Scene；前端兼容无 World 数据的旧角色 |
| SceneChange 堆积过多导致查询变重 | 读取延迟 | 低 | 读取时 LIMIT 20（最近 20 条）；定期归档：超过 100 条的旧记录移入 scene_history_json |
