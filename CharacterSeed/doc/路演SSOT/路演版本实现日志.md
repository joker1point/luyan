### 6.21 — Day4 事件推进系统（v1.5 用户事件）

**代号**：路演版本实现 · Day4  
**目标版本**：v1.5 用户事件  
**核心概念**：为 NPC 角色引入"时间轴"，角色可一天一天推进，每天有日程安排，对话打包为事件，人格随经历变化。

#### 新增功能

| 功能 | 说明 |
|------|------|
| **事件表（Event）** | 角色的"今日待办清单"，每个事件包含 type、content、result_json、order_index、day_number |
| **推进事件** | 完成当前最早的一个 pending 事件，自动将未打包对话打包为 player_dialogue 事件 |
| **迭代一天** | LLM 分析当日完成事件 → 人格变化 + 生成次日日程 + 世界变化 |
| **自动模式** | 一键自动完成当天所有事件并进入下一天 |
| **对话打包** | 一整段聊天被打包为一个"对话事件"，不碎片化 |
| **人格扩展** | 角色新增：说话风格(speaking_style)、核心信念(values)、日常习惯(habits)、长期目标(long_term_goal) |
| **世界变化** | 迭代时 LLM 输出 world_changes，描述世界状态的变化 |

#### 完整工作流

```
创建角色 → Day 1 日程生成
  → 与角色对话（多轮）
  → 点"推进事件" → 对话打包 + 推进第1个日程事件
  → 再点"推进事件" → 推进第2个日程事件
  → 所有事件完成 → 点"迭代一天"
  → LLM 分析 → 人格变化 + 生成 Day 2 日程 + 世界变化
  → 进入 Day 2，重复...
```

#### 架构变更

```
[推进事件] → POST /api/event/advance
  → 打包未打包对话为 player_dialogue Event(completed)
  → 取下一个 pending Event → 标记 completed → 返回 EventResponse

[迭代一天] → POST /api/time/iterate
  → 收集当日全部 completed Event
  → GrowthModule.run() → 人格变化 + 新记忆 + 次日日程 + 世界变化
  → schedule[] → events 表(pending, day_number+1) → day_number += 1

[自动模式] → POST /api/time/auto
  → 循环 advance → advance → ... → iterate
  → 返回 AutoResponse(completed_events[], iterate_result)
```

#### 关键技术决策

1. **对话打包为 player_dialogue 事件**：避免对话碎片化为单个事件，打包后 Growth 以"一段对话"为粒度观测
2. **事件状态机 pending → completed**：不设 active 状态简化管理，对话打包直接标记 completed
3. **Schedule 保底机制**：LLM 输出空 schedule 时自动生成 1 条保底事件，避免"空天"
4. **双接口模式**：手动模式（advance+iterate）用户控制节奏 + 自动模式（auto）一键推演

### 6.22 — Day5 Creation → Growth 工作交接（v1.5 用户事件 · 补完）

**代号**：路演版本实现 · Day5  
**目标版本**：v1.5 用户事件  
**核心概念**：补完 v1.5 缺失的关键环节——Creation LLM 承担 Day 1 初始事件生成职责，作为 Creation → Growth 的工作交接点，确保事件推进循环从角色创建即开始闭环。

#### 修改原因

角色创建后无任何 Day 1 事件，导致"推进事件""迭代一天""自动推演"三个按钮均返回 404（HTTP 业务层异常），当天事件列表为空。本质是 Creation LLM（场景初次确定者）未与 Growth LLM（第三方观察者）对接——Creation 未输出任何事件实体。

#### 修改内容（5 步）

| 步骤 | 文件 | 改动内容 |
|------|------|----------|
| 1️⃣ | `backend/prompts/creation.txt` | 在第 9 字段 `long_term_goal` 后新增第 10 字段 `day1_schedule`，要求 LLM 输出 3-5 条 Day 1 初始事件（含 content/event_type/time_period/order_index），event_type 白名单：`schedule_action/scene_event/character_initiative/player_dialogue`；time_period 白名单：`morning/afternoon/evening/night` |
| 2️⃣ | `backend/services/llm_service.py` | `validate_creation_schema()` 新增 `day1_schedule` 校验逻辑：逐项校验 event_type/time_period/order_index 合法性，跳过空 content 条目，非法值标准化，空数组插入保底事件（与 validate_growth_schema_v2 的保底逻辑完全对称） |
| 3️⃣ | `backend/modules/creation.py` | `CreationModule.run()` docstring 新增 `day1_schedule` 返回值字段说明 |
| 4️⃣ | `backend/main.py` `create_character` | 在 `return db_character` 之前，从 `parsed_data` 提取 `day1_schedule` 逐条调用 `event_crud.create_event` 持久化为 events 表（day_number=1, status=pending）|
| 5️⃣ | `backend/main.py` `auto_advance` | 修复函数签名：`request: AdvanceRequest = Depends()` → `request: AdvanceRequest`，移除非法 `= Depends()` |

#### 完整工作流（修复后）

```
角色创建
  → Creation LLM 产出 day1_schedule（3-5 条事件，含 content/event_type/time_period/order_index）
  → validate_creation_schema 校验 + 保底（非法值标准化，空数组插入 1 条保底事件）
  → create_event 逐条写入 events 表（day_number=1, status=pending）
  → 前端事件列表 → 显示当日待办事件
  → "推进事件" → 逐条推进（pending → completed）不再 404
  → "迭代一天" → Growth 观察 completed 事件 → 人格变化 + 生成 Day 2 schedule
  → "自动推演" → 一键完成全部推进 + 迭代
```

#### 设计决策

1. **Creation LLM 作为事件初始化者**：Creation 设定世界、角色、长期目标，最了解角色首日应发生什么，比 Growth 更适合产出初始日程
2. **格式统一**：`day1_schedule` 字段结构与 Growth `schedule` 字段完全一致（event_type/time_period 共用白名单），两个 LLM 输出同一套数据结构
3. **保底机制**：即使 LLM 完全未输出 `day1_schedule`，校验层自动插入 1 条保底事件，确保不会回到 404 状态
4. **`auto_advance` 独立 Bug**：`= Depends()` 是无参数依赖注入的错误语法，与事件初始化问题独立但同样阻断功能

### 6.23 — v1.6 Phase 1 世界系统具象化

**代号**：路演版本实现 · v1.6 Phase 1  
**目标版本**：v1.6 世界系统  
**核心概念**：将 `world_setting` 从一段静态文本升级为可演化、可共享、可追溯的结构化世界系统。角色从"悬浮在文本描述中"变为"站在具象场景中的存在"。

#### 新增功能

| 功能 | 说明 |
|------|------|
| **World 实体表** | 独立持久化的世界表（name + core_worldview），角色 N:1 归属，为多角色复用预留 |
| **两层场景树** | 概念场景（世界观框架，如大陆/王国/森林）+ 实际场景（角色可互动，如旅店大厅/铁匠铺）。关键约束：实际场景只能归属于概念场景 |
| **SceneChange 迭代表** | 以叙事描述为核心的场景变化记录，分离 character_driven（角色造成）和 external（外部强加）两类变化，保留完整因果链 |
| **Creation 初始化世界** | 创建角色时同步产出 World 行 + 初始概念场景树（覆盖角色当前位置的完整上下层级链）+ 角色起始实际场景 |
| **Growth 迭代世界** | Growth LLM 通过 SceneChange 记录场景变化，同步更新 Scene.description；可新增实际场景（角色发现新地点）或极少数情况下新增概念场景 |
| **9 个世界系统 API** | 世界列表/详情/更新、场景列表/详情/路径/相邻场景、场景变化时间轴 |

#### 核心设计理念

**概念场景 vs 实际场景——按"角色能否直接互动"区分**：

```
概念场景（世界观框架）：
  艾泽拉斯大陆 (conceptual)
    └─ 暴风王国 (conceptual)
         └─ 艾尔文森林 (conceptual)
              └─ 闪金镇 (conceptual)

实际场景（角色可操作空间，归属于概念场景）：
  狮王之傲旅店大厅  → parent: 闪金镇(conceptual)
  闪金镇广场        → parent: 闪金镇(conceptual)
  铁匠铺            → parent: 闪金镇(conceptual)
```

**SceneChange——保留叙事因果链**：不直接修改 Scene.description，而是创建一条变化记录（描述+类型+结构元数据），同步更新 Scene.description 为最新状态，Scene.initial_description 保留 Creation 时的原始描述作为不可变锚点。

**三管线职责边界**：
- Creation：初始化世界（全量骨架 + 局部细节）
- Growth：迭代世界（唯一写入者，通过 SceneChange 间接更新）
- Interaction：只读（读取当前场景路径 + 相邻场景 + 最近变化）

#### 架构变更

```
World 表 (worlds)          ← 独立世界实体，多角色可归属
Scene 表 (scenes)          ← 两层场景树，自引用 parent_scene_id
SceneChange 表 (scene_changes) ← 场景变化叙事因果链
Character 新增字段:
  world_id → worlds.id         ← N:1 归属
  current_scene_id → scenes.id ← 结构化位置，取代纯文本 location
```

### 6.23 — v1.6 Phase 2 交互管线统一事件入口

**代号**：路演版本实现 · v1.6 Phase 2  
**目标版本**：v1.6 世界系统  
**核心概念**：废弃硬编码的事件推进，将日程事件接入 Director+Actor 双 LLM 管线。角色面对事件时拥有"能力集合"——正常回应、主动对话、修改计划、结束事件（成功/超出预期/拖延/失败/跳过）。

#### 新增功能

| 功能 | 说明 |
|------|------|
| **`run_event()` 事件入口** | InteractionPipeline 新增事件模式方法，与对话入口共享同一套 Director+Actor 实例 |
| **能力集合（Capability Set）** | Director 在事件模式下从 4 大能力中选择：respond_normally / initiate_dialogue / modify_plan / complete_event(子类型) |
| **`compute_personality_influence()`** | 纯数学人格加权函数（无 LLM 调用），基于 6 维人格计算 complete_event 子类型概率分布 + 主动对话/修改计划倾向，注入 Director prompt 作为引导 |
| **complete_event 5 子类型** | succeed / exceed / linger / fail / skip——人格加权决定倾向，Director 最终决策。失败≠删除事件，失败事件仍为 completed，其 result 反映失败供 Growth 观察 |
| **事件叙事元数据** | Event 表新增 director_raw / actor_raw / capabilities_applied / emotion / expression 字段，完整保留叙事决策链 |
| **日程可见性** | Director 可看到角色当天完整日程表，支持 modify_plan 能力（调整事件顺序/时间/内容） |

#### 人格加权函数算法

```
complete_event 子类型概率 = 基线概率 ± 人格维度贡献：

succeed_w = 0.30 + courage×0.25 + intelligence×0.20
exceed_w  = 0.08 + intelligence×0.28 + optimism×0.15   // 高智力+乐观 → 超出预期
linger_w  = 0.18 - courage×0.18 + (1-sociability)×0.10  // 低勇气+低社交 → 拖延
fail_w    = 0.12 - intelligence×0.18 - courage×0.10     // 低智力+低勇气 → 搞砸
skip_w    = 0.12 - loyalty×0.22 - empathy×0.12          // 低忠诚+低同理心 → 跳过

其他能力偏置：
dialogue_bias = 0.25 + sociability×0.06 + optimism×0.04
modify_bias   = 0.15 + intelligence×0.04 - loyalty×0.03
```

#### 架构变更

```
advance_event (重写):
  → 打包未打包对话为 player_dialogue Event(completed)
  → 取下一个 pending Event
  → pipeline.run_event() → Director 能力决策 + Actor 叙事生成
  → 持久化 result_json + 元数据
  → 处理 modify_plan / initiate_dialogue callback

auto_advance:
  → 同步使用新 pipeline（检查回调、处理 interleaved 对话事件）
```

| `tests/test_event_pipeline.py` | **NEW** | 42 个事件管线测试（能力组合 + 完成子类型 + 人格加权 + 叙事生成） |

#### 关键设计决策

1. **废弃 EventResolver，复用 InteractionPipeline**：日程事件和玩家对话是同一种"角色刺激"——Director 负责理解上下文决策，Actor 负责生成叙事。割裂导致日程叙事质量远低于对话
2. **能力集合作为有界白名单**：每个能力对应具体的后端实现路径，防止 LLM 产出不可执行的动作
3. **人格加权而非人格决定**：compute_personality_influence() 只提供倾向引导，Director 保有最终决策权
4. **失败≠删除事件**：失败事件仍标记为 completed，其 result 反映失败供 Growth 观察，产生人格影响

### 6.24 — v1.6 场景上下文注入 + 叙事增强（A1/A2 + fix）

**代号**：路演版本实现 · v1.6 叙事增强  
**目标版本**：v1.6 世界系统  
**核心概念**：修复场景上下文在 LLM 管线中的"注入→理解→利用"断链。Growth 产出的日程从"干巴巴的待办清单"升级为"场景感知的叙事脚本"。Actor 生成行为时融入场景元素。前端展示当前世界位置。

#### 问题根因

经过逐行代码排查，发现 v1.6 Phase 1 的场景系统数据已完备，但存在 4 个语义断链：

| # | 位置 | 问题 |
|---|------|------|
| 1 | `growth.txt` | `{scene_context}` 被注入但 prompt 无叙事指令——LLM 产出"下午自习"式的干瘪日程 |
| 2 | `actor.txt` + `interaction.py` | Actor 的 `{scene_context}` 占位符不存在，代码调用时也不传参 |
| 3 | `interaction.py` | `run()` 和 `run_event()` 调用 Actor 时漏传 `scene_context` |
| 4 | `frontend/app.py` | 事件面板只展示裸 content，用户看不到角色身处何地 |

#### 修复内容（4 点）

| 修复点 | 文件 | 改动 |
|--------|------|------|
| **① Growth 叙事指令** | `growth.txt` | 新增"日程生成规则"区块：事件=剧情脚本、融入场景细节、类型多样化（场景事件+角色主动）、场景探索要求、叙事时态（第三人称）、微型故事弧线、好/差日程对比示例 |
| **② Actor 场景占位符** | `actor.txt` | 新增 `{scene_context}` 占位符 + "action 中应体现场景元素"指令 |
| **③ 代码层断链修复** | `interaction.py` | `Actor.generate()` 新增 `scene_context` 参数 → 注入 safe_dict → `generate_with_fallback()` 透传 → `run()` 和 `run_event()` 两处调用传入 |
| **④ 前端场景展示** | `app.py` | 事件列表上方新增"当前场景环境"折叠面板（世界名+场景路径+场景描述）；事件类型图标（🌪️场景/💡主动/💬对话/📋日程）；迭代结果展示场景位置 |

#### 期望表现变化

| 维度 | 修复前 | 修复后 |
|------|--------|--------|
| Growth 日程 | `[afternoon] 下午复习功课` | `午后阳光透过狮王之傲旅店的彩色玻璃窗，{name}在角落旧木桌旁翻开师父留下的地图，试图找到通往矿洞的捷径` |
| 事件类型 | 几乎全是 schedule_action | 含 scene_event（旅店来了受伤的冒险者）+ character_initiative（决定深入矿洞探查） |
| Actor 行为 | "她点了点头" | "身体靠在吧台边，窗外的雨声伴随着她的话语" |
| 前端可见性 | 只有待办事件文本 | 世界路径 + 场景描述 + 事件类型图标 |
