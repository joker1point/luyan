## 一、踩坑记录

### 1. 浏览器缓存导致的 "mode is not defined" 错误

**现象**：
- 从 mock 迁移到 real 模式后，页面报错 "mode is not defined"
- 源码中已无 `mode` 变量引用，但浏览器仍加载旧版 JS chunk

**根因**：
- Vite build 生成的 chunk 文件名带 hash（如 `index-BSpeZXWV.js`）
- 浏览器缓存了旧版 chunk 文件，未加载新 build
- 即使 nginx 配置了 `Cache-Control: no-cache`，浏览器仍可能使用磁盘缓存

**解决方案**：
1. 用户端：`Ctrl+Shift+R` 强制刷新 / DevTools 勾选 "Disable cache" / 无痕模式
2. 服务端：nginx 配置 `Cache-Control: no-cache, no-store, must-revalidate`
3. 开发习惯：build 后验收时必须清缓存或使用无痕模式

**教训**：
> E2E 验收是 build 之外的最后一道防线。build 通过 ≠ 运行时正确。浏览器缓存问题在 SPA 部署时极易被忽视。

### 2. Session Title 乱码（`??□12026-06-27`）

**现象**：
- 侧栏 session 列表显示乱码，如 `??□12026-06-27`
- 数据库中存在脏数据（title 为 `??` 或异常格式）

**根因**：
- `_sanitize_title` 函数仅处理 3+ 个问号（`???+`），漏掉了 2 个问号（`??`）的情况
- 历史脏数据未清理，前端从 localStorage / API 加载时仍显示异常

**解决方案**：
1. 后端 `chat_session_crud.py`：`_sanitize_title` 将问号压缩阈值从 3+ 改为 2+
2. 前端 `useSessions.js`：`_sanitizeTitle` 同步修改
3. 增加"清理后只剩标点返回空串"逻辑
4. 手动修复数据库脏数据（session 106 的 `??` → `新对话`）
5. 前端 bootstrap 时清理 localStorage 缓存脏数据

**教训**：
> 数据清洗函数要考虑边界情况（2个问号、纯标点等）。历史脏数据需主动清理，不能只靠前端防御。

---

### 3. 事件系统出现不存在的角色

**现象**：
- EventsPage 角色筛选器显示"苏晴"（MOCK 角色 ID `char-1`）
- 真实后端只有 ID=2 的"艾拉·万斯"，筛选器失效

**根因**：
- `EventsPage.jsx` 初始化时使用了 `MOCK_CHARACTERS[0]?.id`（`char-1`）
- mock → real 切换后，真实角色列表不包含 MOCK 角色 ID

**解决方案**：
1. `activeCharacterId` 初始值从 `MOCK_CHARACTERS[0]?.id` 改为 `'all'`
2. API 返回真实角色后，检查当前筛选器是否在真实列表中
3. 若当前为 MOCK 角色且不存在于真实列表，重置为 `'all'`

**教训**：
> 前端组件初始化禁止使用 MOCK 数据作为默认值。mock → real 切换时，必须校验状态一致性。

---

### 4. React.memo 别名导致的 ReferenceError

**现象**：
- Vite build 通过，但运行时 `ReferenceError: ReactMemo is not defined`
- 多个文件使用 `ReactMemo`，但只有一个文件加了 import

**根因**：
- `React.memo` 用法需 `import { memo as ReactMemo } from 'react'`
- 多个组件用 `ReactMemo` 别名时，**只在一个文件加 import，其他文件全部 ReferenceError**
- Vite build 是静态分析，不会发现运行时 ReferenceError

**解决方案**：
- 改动后必须 dev server 跑通核心 flow
- E2E 验收是 build 之外的最后一道防线

**教训**：
> 别名 import 容易遗漏。build 通过 ≠ 运行时正确，E2E 验证必不可少。

### 2. jiwen 情绪引擎集成（2026-06-27 完成）

**架构决策**：
- jiwen 是 Node.js 库，CharacterSeed 是 Python FastAPI 项目
- 为避免 HTTP sidecar 的跨进程延迟和运维负担，**1:1 算法移植**到 Python
- 纯数学漂移算法，无 Node 特有 API，可 1:1 移植

**核心文件**：
| 文件 | 职责 |
|------|------|
| `backend/jiwen/jiwen_core.py` | 5 轴漂移算法 + 触发器（~520 行） |
| `backend/jiwen/jiwen_manager.py` | 单例 + DB 持久化 + session_factory 注入 |
| `backend/jiwen/jiwen_scheduler.py` | 后台 tick 调度（asyncio） |
| `backend/modules/memory_decay.py` | Ebbinghaus 衰减 + soft-deprecate |
| `backend/modules/memory_extractor.py` | LLM 提取 5 分区记忆 |
| `backend/modules/summary_trigger.py` | 自适应摘要触发 |
| `backend/api/jiwen_router.py` | 17 个 REST 端点 |

**5 轴状态**：
| 轴 | 范围 | 漂移规则 |
|----|------|----------|
| connection | 0→1 | 基础率 × 加速因子 × 情绪因子 |
| pride | -1→+1 | 0.003/min 回归 0；被冷落时防御性升 |
| valence | -1→+1 | 0.005/min 回归 setpoint |
| arousal | -1→+1 | 0.005/min 回归；想强时升 |
| immersion | 0→1 | 0.01/min 衰减 |

## 三、工程规范总结

### 1. 前端 API 模式切换
- 优先级：URL `?api=real` > localStorage `cs.apiMode.v1` > 默认 `mock`
- 前端通过 `ApiProvider` 包裹 App，使用 `useApi()` 获取正确的 api 对象
- mock → real 切换时，必须校验所有状态一致性（角色 ID、session 等）

### 2. 数据清洗函数
- `_sanitize_title` / `_sanitizeTitle`：将连续 2+ 问号替换为单个问号
- 清理后只剩标点时返回空串
- 历史脏数据需主动清理（数据库 + localStorage）

### 3. 浏览器缓存问题
- build 后验收必须清缓存或使用无痕模式
- nginx 配置 `Cache-Control: no-cache, no-store, must-revalidate`
- SPA 部署时极易忽视浏览器缓存问题

### 4. E2E 验收
- build 通过 ≠ 运行时正确
- E2E 验收是 build 之外的最后一道防线
- 必须 dev server 跑通核心 flow
